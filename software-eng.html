Software Life Cycle: Development Phase
  • Requirement analysis
  • Design
  • Implementation
  • Testing
  • involves a significant input from the stakeholders (future users as well as those with other ties, such as legal or financial interests) of the proposed system.
  • involves creating a plan for the construction of the proposed system.
  • involved the actual writing of programs, creation of data files, and development of databases.
  • essentially equated with the process of debugging programs and confirming that the final software product was compatible with the software requirements specification.
modular implementation the object-oriented paradigm uses objects as the basic modular constituents.
Unified Modeline Language (UML) general-purpose visual modeling language that is intended to provide a standard way to visualize the design of a system.
use case interactions depicted in use case diagram.
use case diagram depicts the proposed system as a large rectangle in which interactions (called use cases) between the system and its users are represented as ovals and users of the system (called actors) are represented as stick figures.
Data relationships:
  • one-to-one
  • one-to-many
  • many-to-many
  • exemplified by the association between patients and occupied private rooms in that each patient is associated with only one room and each room is associated with only one patient.
  • exemplified by the association between physicians and patients in that one physician is associated with many patients and each patient is associated with one (primary) physician.
  • would occur if we included consulting physicians in the physician-patient relationship.
design pattern a predevelopped model for solving a recurring problem in software design.
Software quality assurance (SQA) groups charged with overseeing and enforcing the quality control systems adopted by the organization.
Software testing:
  1. glass box testing
  2. black box testing
  3. beta testing
  4. alpha testing
  1. the software tester is aware of the interior sturcture of the software and uses this knowledge when designing the test.
  2. tests that do not rely on knowledge of the software's interior composition, it is performed from the user's point of view.
  3. a preliminary version of the software is given to a segment of the intended audience with the goal of learning how the software performs in real-life situations before the final version of the product is solidified and released to the market.
  4. similar testing perfomed at the developer's site.
Documentation:
  • user documentation
  • system documentation
  • technical documentation
  • to explain the features of the software and describe how to use them
  • is to describe the software's internal composition so that the software can be maintained later in its life cycle.
  • to describe how a software system should be installed and serviced.
Intellectual property law:
  • software license
  • software disclaimer
  • best defense against software liability claims
  • a legal agreement between the owner and user of a software product that grants the user certain permissions to use the product without transferring ownership rights to the intellectual property.
  • disclaimers in the software license that states the limitations of their liability.
  • to apply sound software engineering principles during the software's development, to use a level of care compatible with the software's application, and to produce and maintain records that validate these endeavors.
